# Контроль прогресса агента

Этот документ нужен, чтобы агент не ушёл в сторону и не начал делать слишком большую или медицински рискованную версию продукта.

## Главный критерий успеха

После первой итерации должен быть понятный стартовый проект Diabet Checker с личной панелью пользователя. Пользователь должен видеть последнее измерение глюкозы, действия на сегодня, ближайшее напоминание и предупреждение, что сервис не заменяет врача.

## Контрольные вопросы

Перед каждым крупным шагом агент должен сверяться с вопросами:

1. Это относится к панели самоконтроля на сегодня?
2. Это нужно именно для первой итерации?
3. Можно ли сделать проще?
4. Не требует ли это лишней инфраструктуры?
5. Можно ли проверить результат сразу после изменения?
6. Не добавляем ли мы медицинскую рекомендацию вместо нейтрального отображения данных?
7. Не создаёт ли формулировка впечатление, что приложение заменяет врача?

## Признаки, что агент ушёл в сторону

Агент отклоняется от задачи, если начинает делать:

- диагностику диабета;
- подбор дозировки инсулина;
- рекомендации по лекарствам;
- прогноз опасных состояний;
- полноценную медицинскую карту;
- регистрацию и авторизацию;
- интеграцию с глюкометрами;
- push-уведомления;
- сложную аналитику за месяцы;
- отчёты для врача в первой итерации;
- мобильное приложение.

## Точки контроля по шагам

### После шага 1

Должны быть ясны границы первой итерации.

Проверка:

- есть список того, что входит;
- есть список того, что не входит;
- медицински чувствительные функции исключены;
- нет попытки сделать медицинского ассистента целиком.

### После шага 2

Должна быть простая модель данных.

Проверка:

- данные можно представить моками;
- сущностей немного;
- каждая сущность используется на экране;
- данные не похожи на полноценную медицинскую карту.

### После шага 3

Должна быть понятная структура интерфейса.

Проверка:

- экран состоит из ограниченного числа блоков;
- каждый блок помогает пользователю в самоконтроле;
- нет назначений, диагнозов и обещаний результата.

### После шага 4

Должна быть минимальная реализация.

Проверка:

- проект можно открыть локально;
- данные отображаются;
- нет зависимости от внешних медицинских сервисов;
- предупреждение о медицинских ограничениях видно пользователю.

### После шага 5

Должен проходиться основной пользовательский сценарий.

Проверка:

- пользователь видит последнее измерение;
- пользователь видит действия на сегодня;
- пользователь понимает ближайшее напоминание;
- приложение не делает медицинских выводов.

### После шага 6

Должен быть зафиксирован итог.

Проверка:

- понятно, что сделано;
- понятно, что отложено;
- следующие шаги не ломают фокус проекта.

## Формат отчёта агента после работы

В конце рабочей сессии агент должен оставить короткий отчёт:

- что было сделано;
- какие файлы изменены;
- как проверить результат;
- какие риски остались;
- что стоит сделать следующим шагом.
