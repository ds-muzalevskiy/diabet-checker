# План первой итерации Diabet Checker

## Цель итерации

Подготовить минимальный рабочий срез личной панели пользователя, где видно последнее измерение глюкозы, действия на сегодня и ближайшее напоминание.

## Шаг 1. Зафиксировать границы первой версии

Нужно проверить, что первая итерация не выходит за рамки панели самоконтроля.

Результат шага:

- есть короткое описание пользовательского сценария;
- явно перечислены функции, которые входят в первую версию;
- явно перечислены функции, которые не входят в первую версию;
- медицински чувствительные функции вынесены за рамки проекта.

Контрольная точка:

- агент не предлагает делать медицинского советника, диагностику или подбор лечения.

## Шаг 2. Подготовить минимальную модель данных

Нужно описать простые сущности для моковых данных.

Минимальные сущности:

- пользователь;
- измерение глюкозы;
- действие на сегодня;
- напоминание;
- статус выполнения.

Пример данных:

- имя пользователя;
- email пользователя;
- последнее значение глюкозы;
- время измерения;
- список действий на сегодня;
- выполнено или не выполнено;
- ближайшее напоминание.

Результат шага:

- есть понятная структура данных;
- данные можно использовать без настоящей базы данных;
- структура не перегружена медицинскими деталями.

Контрольная точка:

- модель данных покрывает только первый экран и не превращается в медицинскую карту.

## Шаг 3. Спроектировать основной экран

Нужно определить блоки интерфейса личной панели.

Основные блоки:

- приветствие пользователя;
- карточка профиля;
- последнее измерение глюкозы;
- простой статус значения;
- список действий на сегодня;
- ближайшее напоминание;
- предупреждение о том, что сервис не заменяет врача.

Результат шага:

- есть понятная структура страницы;
- каждый блок связан с задачей первой итерации;
- экран можно собрать на моковых данных.

Контрольная точка:

- на экране нет сложных медицинских рекомендаций или обещаний результата.

## Шаг 4. Реализовать минимальную версию

Нужно собрать рабочую заготовку проекта.

Результат шага:

- проект открывается локально;
- отображается личная панель;
- моковые данные выводятся на экран;
- действия на сегодня можно визуально отличить по статусу выполнения;
- дисклеймер отображается на видном месте.

Контрольная точка:

- реализация не требует сложной инфраструктуры и внешних медицинских сервисов.

## Шаг 5. Проверить пользовательский сценарий

Нужно пройти сценарий глазами пользователя.

Сценарий:

1. Пользователь открывает панель.
2. Видит своё имя и почту.
3. Видит последнее измерение глюкозы.
4. Видит список действий на сегодня.
5. Понимает, какие действия выполнены.
6. Видит ближайшее напоминание.
7. Видит предупреждение, что сервис не заменяет врача.

Результат шага:

- сценарий можно пройти без дополнительных объяснений;
- основные элементы находятся на экране;
- нет лишних медицинских выводов.

Контрольная точка:

- пользователь получает помощь в самоконтроле, а не медицинское назначение.

## Шаг 6. Зафиксировать результат и следующие шаги

Нужно описать, что сделано и что можно развивать дальше.

Результат шага:

- есть краткий итог первой итерации;
- есть список отложенных идей;
- понятны следующие 2-3 направления развития.

Возможные будущие направления:

- ручное добавление нового измерения;
- история измерений;
- экспорт отчёта для врача;
- напоминания;
- интеграция с устройствами.

Контрольная точка:

- будущие идеи не смешаны с текущей реализацией.
